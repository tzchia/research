01一句話：它是什麼

Ponytail 是一個 plugin——裡面打包了一份 skill 規則集 + 兩個生命週期 hook + /ponytail 系列指令。

它不是函式庫、不是框架，也不會改你的程式碼。它在每一輪對話把那份精簡守則重新注入 agent，逼它在動手之前先停下來問一句：

這段程式，到底需不需要寫？

plugin 這一層

負責「安裝與常駐」：marketplace、hook、/ponytail 指令切換。

skill 這一層

真正的「行為核心」：skills/*/SKILL.md。到只吃指令檔的 host（Cursor/Windsurf）就降級成純規則檔。

02關鍵分辨

Lazy ≠ Careless
懶，不等於草率。

「懶」指的是效率——能不寫就不寫，砍的是過度設計。但有四條紅線永遠不准砍：

① 輸入驗證

信任邊界上的把關

② 錯誤處理

防資料遺失

③ 資安

security 不打折

④ 無障礙

a11y 不犧牲

它砍的是過度設計，不是該做的工。
03運作核心：六階梯（The Ladder）

寫任何程式前，agent 從上往下走這道梯子，停在第一個成立的階梯就動手，不再往下。這是反射動作，不是研究專案。

1這東西需不需要存在？投機需求 → 直接跳過（YAGNI）
2標準函式庫做得到嗎？做得到就用，別手刻 stdlib 已有的
3平台原生功能涵蓋嗎？原生 input ＞ 套件、CSS ＞ JS、DB constraint ＞ 應用層
4已裝的相依套件能解嗎？能就用，絕不為幾行新增相依
5能一行解決嗎？那就一行
6到這步才開始寫寫「剛好能動」的最小程式碼
梯子的精神：第一個能動的懶解法，就是對的解法。比起加程式碼先想能不能刪；比起聰明先求無聊——無聊的程式不會在凌晨三點被叫起來解讀。
04規則：該砍的 vs 紅線

該砍的（懶的部分）

  • 沒被要求的抽象一律不做：沒有單一實作的 interface、沒有只生一種產品的 factory
  • 不寫「為了以後」的 boilerplate / scaffolding
  • 刪除優先於新增，最短可動的 diff 獲勝
  • 遇複雜需求：直接出懶版本，同一則回覆反問「你真需要 X 嗎？」絕不卡住等答案

絕不偷懶的（紅線）

  • 輸入驗證、錯誤處理、資安、無障礙——永遠不上砧板
  • 硬體校準：時鐘會漂、感測器會偏，留下校準旋鈕
  • 兩寫法一樣短時，選邊界情況正確的那個
  • 沒檢查的懶程式 = 未完成：留一個能跑的最小 assert / 小 test
04把「簡單」標記成「刻意」

每一個刻意的簡化，都用 ponytail: 註解標記——讓「簡單」被讀成刻意而非無知。

若這個捷徑有已知天花板（全域鎖、O(n²) 掃描、naïve 啟發法），註解要寫出天花板與升級路徑：

# ponytail: global lock, per-account locks if throughput matters
這就是技術債顯性化的關鍵。後面 /ponytail-debt 指令能把散落各處的 ponytail: 捷徑一網打盡，收成一本帳本——讓「以後」不會變成「永遠不」。
05實例：同模型、同 prompt

任務：「寫一個 Python 函式驗證 email。」（Claude Haiku 4.5，benchmark 真實逐字輸出）

✗ 沒有 Ponytail — 75 行

# regex 版 validate_email…
# …再一個「更 robust」advanced 版
#   (回傳 tuple、長度/local part/
#    雙點檢查…)
# …再一個 email_validator 第三方
#   「production 推薦」版
# …外加三方比較表 + 一段長篇大論

# → 3 種實作、1 張表、N 段散文

✓ 有 Ponytail — 3 行

import re

def is_valid_email(email: str) -> bool:
    return bool(re.match(
      r'^[^@]+@[^@]+\.[^@]+$', email))
75 → 3 行。重點不是字最少，而是只寫任務需要的、且絕不砍驗證。程式碼變小是因為「必要」，不是被硬擠（golfed）。
06三檔強度

用 /ponytail lite|full|ultra|off 切換，預設 full。

lite 輕

照你要求做，但用一行點出更懶的替代，由你決定。

「Done。FYI：functools.lru_cache 一行就能搞定，不用自己養 cache class。」
full 預設

梯子強制執行。stdlib／原生優先、最短 diff、最短解釋。

@lru_cache(maxsize=1000) 掛 fetch。Skipped：自訂 cache，等不夠用再加。
ultra 極端

YAGNI 基本教義派。先丟一行解法，同一口氣質疑剩下需求。

「profiler 開口前不加 cache。手刻 TTL cache 是附帶命中率的 bug 農場。」
07指令
指令作用
/ponytail [lite｜full｜ultra｜off]設定強度或關閉；不帶參數則回報目前等級
/ponytail-review只針對「過度設計」審查目前 diff → 回一張刪除清單
/ponytail-audit同上但掃整個 repo，依「能砍最多」排序
/ponytail-debt把散落的 ponytail: 捷徑收成技術債帳本
/ponytail-help速查表

review／audit 的 finding 標籤（每行：位置、砍什麼、用什麼取代）：

delete:

死碼、用不到的彈性

stdlib:

手刻了 stdlib 已有的

native:

做平台本來就會的事

yagni:

單一實作的抽象

shrink:

同邏輯更少行

結尾只給唯一指標：net: -N lines possible.　沒得砍就 Lean already. Ship.

08架構：薄轉接層 + 共用核心

行為全在 skills/ 與 AGENTS.md；每個 host 專屬檔只是把同一份行為塞進該 agent 的薄轉接層（adapter）。adapter 越薄越好。

ponytail/
├── AGENTS.md              # 26 行核心守則（無 skill 支援時的 always-on 版）
├── skills/               # ★ 真正的行為核心
│   ├── ponytail/        # 懶資深工程師模式（梯子、規則、三檔強度）
│   ├── ponytail-review/ ponytail-audit/ ponytail-debt/ ponytail-help/
├── hooks/               # 兩個 Node.js 生命週期 hook
├── commands/            # /ponytail* 指令（.toml）
├── benchmarks/          # promptfoo 設定 + 可重現實測
└── 各 host 轉接檔        # .claude-plugin/ .codex-plugin/ .cursor/ …
一句話定位：agent 可攜的 skill 散佈包，host 專屬檔只是 adapter。
08它怎麼「常駐」？兩個 hook

Claude Code／Codex 版跑兩個極小的 Node.js hook（所以 node 要在 PATH；不在則安靜略過，不報錯）。

SessionStart
→
跑 ponytail-activate.js，載入目前模式（startup／resume／clear／compact）
UserPromptSubmit
→
每次送出 prompt，跑 ponytail-mode-tracker.js，每一輪重新注入守則，防 agent 飄回過度設計
這就是「成本可能反而變高」的根因：守則每一輪都當 input 重新送進去。在會「想很多」的 reasoning 模型上，這份固定 input + 額外 thinking token，可能比省下的程式碼還貴。（見實測）

10誠實版：跨供應商會翻盤

省錢效果是 Claude 特有的。

同一套測試搬到 OpenAI（10 reps）——在 reasoning 模型上反而更貴：

模型vs baseline延遲正確率
gpt-4.1-mini省 39.6%快 2.5×100%
gpt-5.4-mini貴 26.2%快 1.5×100%
gpt-5.5（最新旗艦）貴 38.7%慢 0.9×100%
原因：reasoning 模型上，每輪重送的守則（大量 input + 額外 reasoning token）超過省下的程式碼。而且 benchmark 只測 single-shot 一輪——真實 session 會每輪重注入，實際成本兩個方向都可能。別把「省 42–75%」當跨模型承諾。
11怎麼看它（評價）

值得用

  • 你主力用 Claude，常被 agent 過度設計、亂裝套件、長篇散文煩到
  • 你要的是可維護性而非炫技——把「少即是多」變成每輪強制反射
  • ponytail: 註解 + /ponytail-debt 把技術債顯性化，比偷偷簡化健康
  • 它不碰你怎麼說話，只管你建什麼，邊界乾淨

留意

  • 省錢非普世：GPT-5.5 等 reasoning 模型上反而更貴更慢
  • 它是 always-on 規則注入，短 prompt 上固定 overhead 可能大於收益
  • 本質是 prompt／skill 工程，不是魔法，效果取決於模型多會聽話
  • 數據是作者自測（可重現），非第三方驗證；但反面數據也誠實寫進 repo，加分
一句總結

Ponytail 把「資深工程師的克制」，
產品化成一份跨 agent 可攜、每輪強制執行的守則。

它最大的價值不是 benchmark 那幾個百分比（會隨模型翻盤），而是把「能不寫就不寫、但紅線不退」這個判斷，從你腦中搬進 agent 的反射裡。
