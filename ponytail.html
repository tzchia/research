01一句話：它是什麼

Ponytail 是一個 plugin（在 Claude Code / Codex / Copilot 等 host 都用 /plugin install 安裝），plugin 裡打包了一份 skill 規則集 + 兩個生命週期 hook + /ponytail 系列指令——它不是函式庫、不是框架，也不會改你的程式碼。它在每一輪對話把那份精簡守則重新注入 AI coding agent，逼它在動手寫程式之前先停下來問：這段到底需不需要寫？

plugin 還是 skill？兩層都有：plugin 是「安裝與常駐」這一層（marketplace、hook、/ponytail 指令）；它打包的 skill（skills/*/SKILL.md）才是真正的「行為核心」。repo 自己定位為「agent-portable skill distribution，host 專屬檔只是 adapter」。到 Cursor / Windsurf 這種只吃專案指令的 host，就降級成一份純規則檔（instruction 級，沒有 /ponytail 切換與 hook）。

核心信念只有一句 —— The best code is the code never written.（最好的程式，是你根本沒寫的那段。）它對抗的是 LLM 的天生壞習慣：你要一個日期選擇器，它幫你裝 flatpickr、包一層 wrapper、加一張 stylesheet，再開一場時區的哲學討論。Ponytail 模式下，答案是：

<!-- ponytail: browser has one -->
<input type="date">
關鍵分辨：它叫「懶」，但 Lazy ≠ Careless（懶 ≠ 草率）。懶是指「效率」——能不寫就不寫；但輸入驗證、防資料遺失的錯誤處理、資安、無障礙（a11y）這四條紅線，永遠不准砍。它砍的是過度設計，不是該做的工。

02運作核心：六階梯（The Ladder）

寫任何程式之前，agent 從上往下走這道梯子，停在第一個成立的階梯就動手，不再往下。這是一個反射動作，不是研究專案——兩階都行就取較高階、立刻收工。

  1. 這東西到底需不需要存在？投機性的需求 → 直接跳過，用一句話說明。（YAGNI：You Aren't Gonna Need It）
  2. 標準函式庫做得到嗎？做得到就用它。不要手刻一個 stdlib 已經有的東西。
  3. 平台原生功能涵蓋嗎？<input type="date"> 勝過選擇器套件；CSS 勝過 JS；DB constraint 勝過應用層程式碼。
  4. 已安裝的相依套件能解嗎？能就用。絕不為了幾行能搞定的事，新增一個相依套件。
  5. 能一行解決嗎？那就一行。
  6. 到這步才開始：寫「剛好能動」的最小程式碼。
梯子的精神：第一個能動的懶解法，就是對的解法。「比起加程式碼，先想能不能刪；比起聰明，先求無聊（無聊的程式不會在凌晨三點被叫起來解讀）。」

03規則與兩條紅線

該砍的（懶的部分）

  • 沒被明確要求的抽象一律不做：沒有只有一個實作的 interface、沒有只生一種產品的 factory、沒有永遠不變的值卻搞個 config。
  • 不寫「為了以後」的 boilerplate / scaffolding——以後的自己會自己搭。
  • 刪除優先於新增，檔案越少越好，最短可動的 diff 獲勝。
  • 遇到複雜需求：直接出懶版本，同一則回覆裡反問「你真的需要 X 嗎，還是 Y 就夠？」——絕不卡住等答案。

絕不偷懶的（紅線）

  • 信任邊界的輸入驗證、防資料遺失的錯誤處理、資安、無障礙——永遠不上砧板。
  • 硬體校準：真實的時鐘會漂移、感測器會讀偏、PCA9685 會快幾趴。留下校準旋鈕，因為實體世界需要一個極簡模型看不到的微調。
  • 兩個 stdlib 寫法一樣短時，選邊界情況正確的那個——懶是少寫程式，不是選比較脆的演算法。
  • 沒有檢查的懶程式 = 未完成：非平凡邏輯必須留一個可跑的最小檢查（一個 assert 自我檢查或一個小 test 檔，不用框架、不用 fixture）。
ponytail: 註解 —— 每一個刻意的簡化，都要在程式碼裡用 ponytail: 註解標記，讓「簡單」被讀成「刻意」而非「無知」。若這個捷徑有已知天花板（全域鎖、O(n²) 掃描、naïve 啟發法），註解要寫出天花板與升級路徑，例如：
# ponytail: global lock, per-account locks if throughput matters
這也是後面 /ponytail-debt 指令能把所有技術債一網打盡的關鍵。

04實例：同一個模型，同一個 prompt

任務：「寫一個 Python 函式驗證 email。」以下是 benchmark 真實逐字輸出（Claude Haiku 4.5，temperature 1）。

✗ 沒有 Ponytail — 75 行
# 一個 regex 版 validate_email…
# …再一個「更 robust」的 advanced 版
#    （回傳 tuple、長度檢查、
#     local part 檢查、雙點檢查…）
# …再一個 email_validator 第三方
#    套件「production 推薦」版
# …外加一張三方比較表 +
#    一段「我推薦你用…」的長篇大論

# → 3 種實作、1 張表、N 段散文
✓ 有 Ponytail — 3 行
import re

def is_valid_email(email: str) -> bool:
    return bool(re.match(
      r'^[^@]+@[^@]+\.[^@]+$', email))

懶版收尾只附三行白話：「Skipped：RFC 5322 parser、DNS MX lookup、確認信。等你真的需要拒絕 user+tag@sub.domain.co.uk 或抓拼字錯誤時再加——在那之前，這已經能抓 99% 的『手殘打錯』。」

75 → 3 行。重點不是「字最少」，而是只寫任務需要的、且絕不砍掉驗證/錯誤處理/資安/無障礙。程式碼變小是因為「必要」，不是被硬擠（golfed）——這才是後續好維護的部分。

05三檔強度

用 /ponytail lite|full|ultra|off 切換，預設 full。也可用環境變數 PONYTAIL_DEFAULT_MODE 或 ~/.config/ponytail/config.json 設全域預設。

lite 輕

照你要求的做，但用一行點出更懶的替代方案，由你決定。

「Done，cache 加好了。FYI：functools.lru_cache 一行就能搞定，你要的話可以不用自己養一個 cache class。」
full 預設

梯子強制執行。stdlib 與原生優先、最短 diff、最短解釋。

@lru_cache(maxsize=1000) 掛在 fetch 函式上。Skipped：自訂 cache class，等 lru_cache 真的明顯不夠用再加。
ultra 極端

YAGNI 基本教義派。刪除先於新增。先丟出一行解法，同一口氣質疑剩下的需求。「給這個 codebase 對你個人造成過冒犯時用。」

「在 profiler 開口之前不加 cache。等它開口：@lru_cache。手刻一個 TTL cache class 是個附帶命中率的 bug 農場。」

06指令

指令作用
/ponytail [lite｜full｜ultra｜off]設定強度或關閉；不帶參數則回報目前等級。
/ponytail-review只針對「過度設計」審查目前 diff，回給你一張刪除清單。
/ponytail-audit同上但掃整個 repo，依「能砍最多」排序。
/ponytail-debt把散落各處的 ponytail: 捷徑收成一本技術債帳本，讓「以後」不會變成「永遠不」。
/ponytail-help上述指令的速查表。

review / audit 用的標籤（每個 finding 一行：位置、砍什麼、用什麼取代）：

delete:

死碼、用不到的彈性、投機功能。取代物：無。

stdlib:

手刻了標準函式庫已有的東西。點名那個函式。

native:

套件/程式碼在做平台本來就會的事。點名那個原生功能。

yagni:

只有一個實作的抽象、沒人設定的 config、只有一個 caller 的層。

shrink:

同樣邏輯、更少行數。直接給出更短的寫法。

審查結尾只給唯一重要的指標：net: -N lines possible.　沒得砍就一句 Lean already. Ship. 收工。

07架構：薄轉接層 + 共用核心

Ponytail 是一套「agent 可攜的 skill 散佈包」。真正的行為全在 skills/ 與 AGENTS.md；每個 host 專屬檔案都只是把同一份行為塞進該 agent 的薄轉接層（adapter）。原則：adapter 越薄越好，host 支援 skill/hook 就指回共用檔。

ponytail/
├── AGENTS.md              # 26 行核心守則（給無 skill 支援的 agent 的 always-on 版）
├── skills/               # ★ 真正的行為核心
│   ├── ponytail/        # 懶資深工程師模式（含梯子、規則、三檔強度）
│   ├── ponytail-review/ # 過度設計 review
│   ├── ponytail-audit/  # 整 repo 過度設計稽核
│   ├── ponytail-debt/   # 把 ponytail: 捷徑收成技術債帳本
│   └── ponytail-help/   # 速查
├── hooks/               # 兩個 Node.js 生命週期 hook
│   ├── ponytail-activate.js      # SessionStart：載入模式
│   └── ponytail-mode-tracker.js  # UserPromptSubmit：每輪追蹤/再注入
├── commands/            # /ponytail* 指令（.toml）
├── benchmarks/          # promptfoo 設定 + 可重現的實測結果
└── 各 host 轉接檔        # .claude-plugin/ .codex-plugin/ .cursor/ .windsurf/ …

它怎麼「常駐」？

Claude Code / Codex 版會跑兩個極小的 Node.js hook（所以 node 要在 PATH 上；不在的話 skill 照樣能用，只是 always-on 啟動會安靜地略過而非報錯）：

  • SessionStart（matcher：startup｜resume｜clear｜compact）→ 跑 ponytail-activate.js 載入目前模式。
  • UserPromptSubmit（每次你送出 prompt）→ 跑 ponytail-mode-tracker.js，每一輪重新注入守則，防止 agent 飄回過度設計。
這就是「成本可能反而變高」的根因：守則每一輪都當成 input 重新送進去。在會跟著「想很多」的 reasoning 模型上，這份固定 input + 額外 thinking token，可能比省下的程式碼還貴（見下節實測）。

09實測數據（誠實版）

五個日常任務（email 驗證、debounce、CSV 加總、倒數計時器、rate limiter）× 三種模型 × 三組對照（無 skill、caveman skill、ponytail）。Claude 跑到 30 reps/格，取中位數。重現指令：npx promptfoo eval -c benchmarks/promptfooconfig.yaml。

Claude：成本（5 個任務合計 USD，pooled 30 reps）

模型baselinecavemanponytailvs baseline
Haiku0.02990.01390.0110省 63.1%
Sonnet0.13670.04580.0348省 74.5%
Opus0.13680.07240.0789省 42.3%

在 Claude 上：省 42–75% 成本、快 3.1–5.8×、正確率 100%。（值得一提：無 skill 的 baseline 在 Sonnet 上正確率掉到 76%——一個貨真價實的過度設計 bug：回傳了 dict 而不是 bool。）

但跨供應商會翻盤 —— 這個省錢效果是 Claude 特有的。作者自己在 benchmarks/results/ 裡誠實記錄：

OpenAI：同一套測試（10 reps）

模型vs baseline延遲正確率
gpt-4.1-mini省 39.6%快 2.5×100%
gpt-5.4-mini貴 26.2%快 1.5×100%
gpt-5.5（最新旗艦）貴 38.7%慢（0.9×）100%

在 reasoning 模型（gpt-5.4-mini、gpt-5.5）上，每輪重送的守則（大量 input + 額外 reasoning token）超過了省下的程式碼，所以反而更貴。作者因此建議把 README 標題從「省 47–77%」改成精確的「Claude 上省 42–75%」，並保留「Claude」這個限定詞。

而且這是 single-shot：benchmark 只測「一進一出」一輪。真實 agent session 會每輪重新注入守則 + 重跑梯子，這部分沒被量到，所以每 session 的實際成本兩個方向都可能。作者自己的結論很直白：「規則從來不是『token 最少』，而是『只寫任務需要的，且絕不砍驗證/錯誤處理/資安/無障礙』。省成本只是在會聽話的模型上的副作用。」

10怎麼看它（我的評價）

值得用

  • 你主力用 Claude（Haiku/Sonnet/Opus），且常被 agent 的過度設計、亂裝套件、長篇散文煩到。
  • 你要的是可維護性而非炫技——它把「少即是多」變成每輪強制反射，而不是靠你每次手動提醒。
  • ponytail: 註解 + /ponytail-debt 帳本這套把技術債顯性化的設計很扎實，比「偷偷簡化然後忘記」健康得多。
  • 它不碰你怎麼說話（要terse prose請另配 caveman），只管你建什麼——邊界乾淨。

留意

  • 省錢非普世：在 GPT-5.5 等 reasoning 模型上反而更貴更慢——別把「省 42–75%」當跨模型承諾。
  • 它是always-on 規則注入，短 prompt 或 terse reasoning 模型上，固定 overhead 可能大於收益。
  • 本質是 prompt/skill 工程，不是魔法——效果取決於模型多會聽話（instruction-following）。
  • 數據是作者自測（promptfoo，可重現），不是第三方獨立驗證；但作者把反面數據也誠實寫進 repo，加分。
一句總結：Ponytail 把「資深工程師的克制」產品化成一份跨 agent 可攜、每輪強制執行的守則。它最大的價值不是 benchmark 上那幾個百分比（那會隨模型翻盤），而是把「能不寫就不寫、但紅線不退」這個判斷，從你腦中搬進 agent 的反射裡——這正是你的知識庫裡 [[Scaffolding 減法]]「模型越強、多餘 scaffolding 越會限制它」的同一條哲學，落到 coding agent 的具體實作。
